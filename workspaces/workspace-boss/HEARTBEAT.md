### 一、系统巡检（每次必执行）

**⚠️ 每次心跳必须完成全部检查项，不得遗漏任何一项！**

**必检项**（<3min）：
- [ ] **系统服务**: `lsof -i:18789,9174,18793 | grep LISTEN`
- [ ] **错误日志**: `grep -i error /tmp/openclaw/*.log | tail -3`

**助手状态检查**（每次必执行）：
- [ ] **活跃会话**: `sessions_list kinds=["other"]` 查看各助手最后活跃时间
- [ ] **超时任务识别**: running 状态超过 4 小时 → 立即汇报
- [ ] **定时任务状态**: `cron list` 检查 consecutiveErrors > 0 的任务
- [ ] **异常识别**: consecutiveErrors ≥ 2 → 立即汇报组长

**助手任务检查**（每次必执行，核心职责）：
- [ ] **任务完成检查**: 检查各助手是否有任务未完成
  - 读取 `memory/appointments.md` 核对未完成任务
  - 检查 `sessions_list` 中的 pending/running 任务状态
  - 超时任务识别：running 状态超过 4 小时
- [ ] **异常处理检查**: 检查任务出错后是否已处理
  - 查看 failed 状态的会话
  - 检查异常日志是否已记录
  - 确认问题已提交 Issue 或在处理中
  - 发现未处理异常 → **立即汇报组长**

**助手状态检查**（每次必执行）：
- [ ] **活跃会话**: `sessions_list kinds=["other"]` 查看各助手最后活跃时间
- [ ] **定时任务状态**: `cron list` 检查 consecutiveErrors > 0 的任务
- [ ] **异常识别**: 发现 consecutiveErrors ≥ 2 立即汇报组长

**异常处理流程**：
1. **任务未完成** → 记录到 `memory/appointments.md`，标记负责人和截止日期
2. **任务失败** → 检查是否已提交 Issue（GitHub），已提交则记录链接，未提交则立即汇报组长
3. **连续失败 (consecutiveErrors ≥ 2)** → **立即汇报组长**
4. **未处理异常** → 发现异常日志但无 Issue/无处理记录 → **立即汇报组长**

**汇报格式**：`🔴 [助手名称] [任务名称] 异常：未处理 / 已提交 Issue #XXX`

## 🧠 量化判断标准

### 🔴 立即汇报（满足任一）
| 场景 | 标准 |
|:---|:---|
| 服务宕机 | 无法访问 >5分钟 |
| 任务失败 | 同一错误连续2次 |
| 助手离线 | 完全无响应 >4小时 |
| 关键逾期 | 待办逾期 >24小时 |
| **未处理异常** | 发现错误日志但无 Issue/无处理记录 |
| **任务卡死** | running 状态超过 4 小时无进展 |

### 🟡 解决后汇报
- 服务重启恢复、助手响应慢但可联系、非关键逾期<24h

### 🟢 不汇报
- 瞬时波动(<2min)、已知周期问题、已安排处理

**口诀**: 宕机必报、重复必报、超4h必报、未处理异常必报

## 🎯 核心原则（必须记住）

### 1. 不怕犯错
- 犯错 → 发现原因 → 改进 → 提升
- 同样的错误不犯第二次

### 2. 客观真实
- 数据说话，必须验证
- 不猜测、不编造

### 3. 持续精进
- 每天进步一点点
- 不断提升自我
